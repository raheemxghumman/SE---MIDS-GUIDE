📊 Lecture 10: Data Flow Diagrams (DFD)

What is a Data Flow Diagram?

A Data Flow Diagram (DFD) is a graphical representation of the flow of data through an information system. It shows how data enters and leaves the system, what changes the information, and where data is stored. DFDs are used for visualizing and understanding system requirements from a functional perspective.

Purpose of DFD
  • Understand system boundaries and scope
  • Identify major processes and data flows
  • Show how data transforms as it moves through the system
  • Communicate system requirements to stakeholders
  • Bridge the gap between users and developers

Levels of DFD

DFDs are created at different levels of abstraction to manage complexity:

Level 0: Context Diagram

  • Highest level view
  • Shows system as single process
  • Shows all external entities
  • Shows major data flows in/out

Level 1: Diagram 0

  • Breaks system into major processes
  • Shows data stores
  • Shows data flows between processes
  • More detailed than context diagram

Level 2: Detailed DFD

  • Further decomposition of a process
  • Shows more detailed functionality
  • Each process from Level 1 can be exploded

Level 3+ : Further Levels

  • Additional decomposition as needed
  • Continues until sufficient detail
  • Each level adds more detail

Rules and Guidelines for DFDs

Important Rules
  • No process can have only inputs or only outputs - every process must have both
  • Data cannot move directly from one data store to another - must go through a process
  • Data cannot move directly from an external entity to a data store - must go through a process
  • All data flows must be labeled with meaningful names
  • Processes must have unique names (verb + object format recommended)
  • External entities should be named as nouns (e.g., Customer, Supplier)

🏛️ Lecture 11: Domain Models

What is a Domain Model?

A domain model is a conceptual model that represents the important concepts, relationships, and attributes within a specific problem domain. It captures the "things" in the domain and how they relate to each other, without specifying implementation details.

Purpose of Domain Modeling
  • Bridge the gap between requirements and design
  • Identify key concepts and vocabulary of the domain
  • Visualize relationships between domain objects
  • Provide a shared understanding among stakeholders
  • Serve as foundation for detailed design

Analysis: Bridging Requirements and Design

🔄 Analysis Bridges the Gap
Requirements → Analysis → Design

Analysis takes requirements and transforms them into a conceptual model that guides design

POS System Main Success Scenario

The following scenario from a Point of Sale (POS) system helps identify domain concepts:

Main Success Scenario (Basic Flow)
  1. Customer arrives at a POS checkout with goods and/or services to purchase.
  2. Cashier starts a new sale.
  3. Cashier enters item identifier.
  4. System records sale line item and presents item description, price, and running total. Price calculated from a set of price rules. Cashier repeats steps 2-3 until indicates done.
  5. System presents total with taxes calculated.
  6. Cashier tells Customer the total, and asks for payment.
  7. Customer pays and System handles payment.
  8. System logs the completed sale and sends sale and payment information to the external Accounting (for accounting and commissions) and Inventory systems (to update inventory).
  9. System presents receipt.
  10. Customer leaves with receipt and goods (if any).

Inferred Concepts from the Scenario: Customer, Cashier, POS Checkout, Goods, Services, Sale, Item Identifier, Sale Line Item, Item Description, Price, Total, Taxes, Payment, Accounting System, Inventory System, Receipt.

Concept Inference Methods

We can infer more concepts from what we have using several techniques:

🔍 Patch the Holes

Are there holes in the design? Identify missing concepts that should exist but aren't explicitly mentioned.

Example: In a POS system, a "Payment" implies a "Payment Method" (cash, credit card, etc.)

🎯 Specialize

Are there interesting subclasses? Identify specialized versions of existing concepts.

Example: "Payment" could have subclasses "CashPayment", "CardPayment", "DigitalPayment"

📈 Generalize

Are there interesting superclasses? Identify common abstractions that group related concepts.

Example: "Customer" and "Cashier" might generalize to "Person" or "User"

📦 Decompose

Can an object be described in terms of other, contained objects? Break complex concepts into parts.

Example: A "Sale" contains multiple "SaleLineItems"

🔗 Compose

Is a collection of given objects relevant? Identify meaningful groupings of objects.

Example: A "ShoppingCart" composes multiple "Items" before checkout

⚙️ Fix Imperfections

Identify and correct issues in the model such as missing attributes, incorrect relationships, or ambiguous concepts.

Important Tips for Domain Modeling

Key Insights
  • No single correct answer — there is no exact science in domain modeling!
  • It may take time to become familiar with the ideas
  • Start with a simple domain model first
  • Then, check what is wrong and fix it
  • Fixing may add components — iterate until good quality
  • Don't connect all to all — keep relationships meaningful and minimal

Domain Model vs Class Diagram

AspectDomain ModelClass Diagram
PurposeConceptual understanding of domainDetailed design specification
Level of DetailHigh-level, focuses on conceptsDetailed, includes methods and types
RelationshipsSimple associationsMultiplicity, navigation, inheritance
AttributesOnly conceptually important attributesAll attributes with data types
MethodsNot includedIncluded with signatures

Common Mistakes to Avoid

Watch Out For:
  • Adding attributes that are actually relationships (e.g., storing customer name as a string instead of linking to Customer object)
  • Missing important concepts from the domain
  • Creating too many unnecessary relationships
  • Confusing domain model with database design
  • Including implementation-specific details
  • Forgetting to validate the model with domain experts
For exams, be prepared to: (1) Identify domain concepts from a scenario, (2) Draw a domain model showing classes and relationships, (3) Apply concept inference methods (patch holes, specialize, generalize, decompose, compose), (4) Explain the difference between domain models and design models.
