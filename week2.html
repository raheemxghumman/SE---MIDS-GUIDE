Lecture 3‑4: Software Development Life Cycle (SDLC) and Process Models

SDLC Definition

The Software Development Life Cycle (SDLC) is a well‑defined, structured sequence of stages in software engineering to develop the intended software product efficiently. It provides a series of steps to be followed to design and develop a software product.

Definition: SDLC

A framework that describes the activities performed at each stage of a software development project, from initial concept through deployment and maintenance.

SDLC Phases (10 Phases)

The SDLC typically consists of ten distinct phases, though the exact number can vary depending on the source.

  1. Communication — User initiates request for software product, negotiates terms, and submits request in writing.
  2. Requirement Gathering — Collect requirements via observation, studying existing systems, questionnaires, and interviews.
  3. Feasibility Study — Analyze if software can fulfill all user requirements; segregate into user, functional, and system requirements.
  4. System Analysis — Decide roadmap and choose suitable software model; understand limitations, system problems, and organizational impact.
  5. Software Design — Combine requirements and analysis to produce logical and physical designs (data dictionaries, diagrams, pseudo‑code).
  6. Coding — Implementation of design by writing program code in a suitable programming language.
  7. Testing — Conduct thorough testing at various levels (module, program, product, in‑house, user) to discover errors early.
  8. Integration — Integrate software with libraries, databases, and other external programs.
  9. Deployment — Install software on user machines; test for portability and adaptability.
  10. Operation and Maintenance — Ensure efficient operation with fewer errors; provide training and update code as environment changes.

Process Models Overview

The main process models covered are:

  • Waterfall Model
  • V Model
  • Incremental Model
  • Evolutionary Model (Prototyping)
  • Spiral Model

Waterfall Model

The waterfall model, sometimes called the classic life cycle, suggests a systematic, sequential approach to software development. It was the first process model to be introduced and followed widely in software engineering. The whole process is divided into separate phases in linear order.

Advantages

  • Simple and easy to understand and use
  • Easy to manage – each phase has specific deliverables and a review process
  • Phases are processed and completed one at a time; phases do not overlap
  • Works well for smaller projects where requirements are clearly defined and well understood

Disadvantages

  • Real projects rarely follow the sequential flow; changes can cause confusion
  • Difficult for customers to state all requirements explicitly; model cannot accommodate natural uncertainty
  • Customer must have patience; working version is not available until late in the project
  • Linear nature leads to "blocking states" where team members must wait for dependent tasks
  • Once in testing stage, it is very difficult to go back and change something not well thought out
  • No working software is produced until late in the life cycle
  • High amounts of risk and uncertainty
  • Not good for complex and object‑oriented projects
  • Poor model for long and ongoing projects
  • Not suitable for projects where requirements are at moderate to high risk of changing
When to Use Waterfall

Requirements are very well known, clear, and fixed; product definition is stable; technology is understood; there are no ambiguous requirements; good resources with required expertise are available freely; the project is short.

V Model

As a software team moves down the left side of the V, basic problem requirements are refined into progressively more detailed and technical representations. Once code has been generated, the team moves up the right side of the V, performing a series of tests (quality assurance actions) that validate each of the models created. In reality, there is no fundamental difference between the classic life cycle and the V‑model. The V‑model provides a way of visualizing how verification and validation actions are applied to earlier engineering work.

Example

Hospital System

The left side of the V defines requirements and design, while the right side validates each level through unit testing, integration testing, system testing, and acceptance testing.

Incremental Model

Also known as the Successive Version model. First, a simple working system implementing only a few basic features is built and delivered to the customer. Then many successive iterations/versions are implemented and delivered until the desired system is released.

Life Cycle Activities
  • Requirements are broken down into several modules that can be incrementally constructed and delivered
  • At any time, the plan is made just for the next increment, not for long‑term plans
  • Development team first undertakes to develop core features (those that do not need services from other features)
  • Once core features are fully developed, they are refined to increase levels of capability by adding new functions in successive versions
  • Each incremental version is usually developed using an iterative waterfall model
  • Customer feedback is taken and incorporated into the next version
  • Each version has more additional features over previous ones
Example

Modules A, B, C

Version 1 has only module A. Version 2 has A and B. Version 3 has A, B, and C.

Evolutionary (Prototyping) Model

Prototyping is the process of quickly putting together a working model (a prototype) to test various aspects of a design, illustrate ideas or features, and gather early user feedback. Prototyping is often treated as an integral part of the system design process, believed to reduce project risk and cost. The prototyping model suggests that before carrying out development of actual software, a working prototype of the system is built.

What is a Prototype?

A prototype is a toy implementation of a system having limited functional capabilities, low reliability, and inefficient performance. Its purpose is to illustrate input data formats, reports, and interactive dialog to the customer.

Advantages of Prototyping

  • Flexible in design
  • Easy to detect errors
  • Can find missing functionality easily
  • Scope for refinement; new requirements can be easily accommodated
  • Ensures greater level of customer satisfaction and comfort
  • Helps developers and users both understand the system better

Disadvantages of Prototyping

  • Costly
  • Poor documentation because of continuously changing customer requirements
  • Too much variation in requirements possible
  • Customers sometimes demand actual product to be delivered soon after seeing an early prototype
  • Incomplete or inadequate problem analysis possible
  • May increase complexity of the system

Spiral Model

The spiral model is one of the most important SDLC models, providing support for Risk Handling. In its diagrammatic representation, it looks like a spiral with many loops. The exact number of loops is unknown and can vary from project to project. Each loop of the spiral is called a Phase of the software development process. The project manager dynamically determines the number of phases depending upon project risks.

Risk Handling in Spiral Model

A risk is any adverse situation that might affect successful completion of a software project. The most important feature of the spiral model is handling unknown risks after the project has started. Such risk resolutions are easier done by developing a prototype. The spiral model supports coping with risks by providing scope to build a prototype at every phase.

Four Quadrants of Each Spiral Phase

  1. Objectives determination and identify alternative solutions — Requirements are gathered from customers, objectives are identified, elaborated, and analyzed at the start of every phase. Alternative solutions possible for the phase are proposed.
  2. Identify and resolve Risks — All possible solutions are evaluated to select the best possible solution. Risks associated with that solution are identified and resolved using the best possible strategy. At the end of this quadrant, a prototype is built for the best possible solution.
  3. Develop next version of the Product — Identified features are developed and verified through testing. At the end of this quadrant, the next version of the software is available.
  4. Review and plan for the next Phase — Customers evaluate the so‑far developed version of the software. Planning for the next phase is started.

Advantages of Spiral Model

  • Risk Handling — Projects with many unknown risks are best handled using this model due to risk analysis and handling at every phase
  • Good for large projects — Recommended for large and complex projects
  • Flexibility in Requirements — Change requests at later phases can be incorporated accurately
  • Customer Satisfaction — Customer can see development at early phases and becomes habituated with the system

Disadvantages of Spiral Model

  • Complex — Much more complex than other SDLC models
  • Expensive — Not suitable for small projects
  • Too much dependability on Risk Analysis — Without highly experienced experts, project is likely to fail
  • Difficulty in time management — Number of phases is unknown at the start, so time estimation is very difficult

Lecture 5‑6: Agile Software Development

Rapid Software Development – Why It Matters

Rapid development and delivery is now often the most important requirement for software systems. Businesses operate in a fast‑changing requirement environment, and it is practically impossible to produce a set of stable software requirements. Software has to evolve quickly to reflect changing business needs. Rapid software development processes are designed to produce useful software quickly. The software is not developed as a single unit but as a series of increments, with each increment including new system functionality.

Fundamental Characteristics of Agile Process

  • Specification, design, and implementation are interleaved
  • No detailed system specification
  • Design documentation is minimized
  • User requirements document only defines important characteristics of the system
  • System is developed as a series of versions with stakeholders involved in version evaluation
  • Users propose changes and new requirements for later versions
  • User interfaces are often developed using an IDE and graphical toolset

Agile Manifesto (Four Core Values)

Agile Manifesto

Individuals and interactions over processes and tools
Working software over comprehensive documentation
Customer collaboration over contract negotiation
Responding to change over following a plan

Agile Method Applicability

Agile methods are applicable for:

  • Product development where a software company is developing a small or medium‑sized product for sale
  • Custom system development within an organization where there is clear commitment from the customer to become involved
  • Projects where there are not a lot of external rules and regulations affecting the software
Problems with Agile Methods
  • Difficulty keeping the interest of customers involved in the process
  • Team members may be unsuited to intense involvement
  • Prioritizing changes can be difficult where there are multiple stakeholders
  • Contracts may be a problem (requirements document is part of contract)
  • Organizations that have spent years formalizing processes find it difficult to adopt informal process models

Extreme Programming (XP)

Perhaps the best‑known and most widely used agile method. XP takes an 'extreme' approach to iterative development:

  • New versions may be built several times per day
  • Increments are delivered to customers every 2 weeks
  • All tests must be run for every build, and the build is only accepted if tests run successfully
XP Practices
  • Planning – Customer writes user stories; team breaks them into tasks.
  • Design – Uses CRC cards, spikes, and refactoring.
  • Coding – Pair programming, test‑first development.
  • Testing – Automated tests, user involvement in validation.
CRC Card Example (Student Registration System)

Class: Student

Responsibilities: Manage student profile information, enrol in courses, view registration status, drop or modify course enrolment
Collaborators: Course, Registration, Department, Payment

Test‑first development follows four steps: Write the test, run the test (initially fails), write code to pass the test, refactor code.

Scrum

Scrum is an iterative and incremental Agile framework focusing on managing iterative development.

Three Phases of Scrum

  1. Outline planning phase – Establish general objectives and design software architecture
  2. Sprint cycles – Series of cycles (2‑4 weeks each) developing an increment
  3. Project closure phase – Wrap up project, complete documentation, assess lessons learned

Key Roles

  • Scrum Master – Facilitator, arranges daily meetings, tracks backlog
  • Product Owner – Manages Product Backlog, makes final decisions
  • Scrum Team – Self‑organizing team that develops the product
  • Customer – Participates in backlog items
  • Management – Final decision making, selects Product Owner

The Sprint Cycle (Four Phases)

  1. Assess Phase – Review product backlog, assign priorities and risks
  2. Selection Phase – Select features to be developed during the sprint
  3. Develop Phase – Team develops software, isolated from distractions
  4. Review Phase – Present work to stakeholders, gather feedback
Daily Scrum Meeting (15 minutes)

Each team member answers three questions:

  • What did you do since the last team meeting?
  • What obstacles are you encountering?
  • What do you plan to accomplish by the next team meeting?

Kanban Software Development

Kanban is an Agile workflow management method that visualizes work and focuses on continuous delivery without overloading the team. Core idea: See the work, limit work in progress (WIP), and improve flow.

Key Principles of Kanban
  • Visualize Work – Use a Kanban board to track progress
  • Limit Work in Progress (WIP) – Prevent task overload
  • Focus on Flow – Continuously improve processes and remove bottlenecks
  • Explicit Policies – Clear guidelines for processes
  • Continuous Improvement – Regular feedback and adjustments
Kanban Board Columns

Backlog → Analysis → Development → Test Board → Deploy Task → Done

Key Takeaways
  • No single best model fits all projects
  • High‑risk systems → Spiral / V‑Model
  • Changing requirements → Agile
  • Phased delivery → Incremental
  • Legal / compliance‑heavy → Waterfall / V‑Model

Exam Tips for SDLC and Agile

Memorize the 10 SDLC phases in order (Communication, Requirement Gathering, Feasibility Study, System Analysis, Software Design, Coding, Testing, Integration, Deployment, Operations & Maintenance).
Know which model to choose for which scenario: Waterfall (clear, fixed requirements), V‑Model (safety‑critical), Incremental (large system with modules), Prototyping (unclear requirements), Spiral (high‑risk, large projects), Agile/XP/Scrum (changing requirements), Kanban (continuous flow).
Agile Manifesto four values – memorize exactly as written.
XP Practices – planning, design (CRC cards, spikes, refactoring), coding (pair programming, test‑first), testing (automated, JUnit).
Scrum roles – Scrum Master, Product Owner, Scrum Team, Customer, Management – know responsibilities of each.
Sprint cycle phases – Assess, Selection, Develop, Review.
Daily Scrum three questions – what did you do, what obstacles, what will you do.
Kanban principles – visualize work, limit WIP, focus on flow, explicit policies, continuous improvement.
