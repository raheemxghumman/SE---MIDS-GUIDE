What is a Use Case?

Definition

A requirements analysis concept that describes the system's actions from the user's point of view. It's a case of a use of the system/product that tells a story—a sequence of events involving interactions of a user with the system.

Key Characteristics

  • Specifies one aspect of the system's behavior without specifying its structure
  • Oriented toward satisfying a user's goal
  • Describes a sequence of events involving interactions between user and system
  • Provides a high-level view of behavior observable to someone outside the system
  • Represents a single unit of meaningful work

Use Case Terminology

TermDefinition
ActorSomething with a behavior or role (person, another system, organization)
ScenarioA specific sequence of actions and interactions between actors and the system (also called use case instance)
Use CaseA collection of related success and failure scenarios, describing actors using the system to support a goal

Actors

What is an Actor?

An actor represents roles which may include human users, external hardware, or other systems. Include all user roles that interact with the system and system components that initiate or trigger a use case.

Scenarios

A scenario is a specific sequence of actions and interactions between actors and the system. It's also known as a use case instance.

Example Scenario

Customer Makes a Purchase

  1. Customer browses products
  2. Customer adds items to shopping cart
  3. Customer proceeds to checkout
  4. System calculates total with taxes
  5. Customer enters payment information
  6. System processes payment
  7. System confirms order and sends receipt

User Stories

What is a User Story?

An abbreviated description of a use case used in agile development. It answers three questions: WHO, WHAT, and WHY.

User Story Format

User Story Template

As a [WHO - type of user],
I want [WHAT - some goal],
So that [WHY - some reason/benefit].

Use Case Diagrams (UCD)

What is a Use Case Diagram?

A UML diagram that presents interactions between the system and other systems or with the system's users (actors). It describes who will use the system and the ways in which a user will interact with the system to achieve a certain goal.

Include Relationship «include»

«include»

One use case (base) includes the functionality of another (inclusion case). Used to support reuse of functionality. A base use case MUST call the included use case.

📌 «include» Relationship - MANDATORY
Login
«include»
Authenticate User

Login MUST include Authenticate User → Always executed

Extend Relationship «extend»

«extend»

One use case (extension) extends the behavior of another (base) under certain conditions. Think of if-then decision points.

🔀 «extend» Relationship - OPTIONAL
Process Order
«extend»
(if coupon present)
Apply Discount

Apply Discount ONLY extends Process Order IF customer has a coupon

Generalization Relationship

📈 Generalization (Inheritance)
Registration
▲
Student Registration Faculty Registration

Child use cases inherit behavior from parent and can add/override

«include» vs «extend»

«include»:

  • MUST execute
  • Always happens
  • Mandatory relationship
  • Supports reuse

 

«extend»:

  • MAY execute
  • Conditional behavior
  • Optional relationship
  • Handles variations

Written Examples

ATM System Example

Actors:

  • Customer
  • Bank System

Use Cases:

  • Withdraw Cash
  • Deposit Money
  • Check Balance
  • Transfer Funds
  • Authenticate User (included in all transactions)
  • Print Receipt (extends all transactions)

Relationships:

  • All transaction use cases «include» Authenticate User
  • Print Receipt «extends» Withdraw Cash (optional)
  • Bank System actor is associated with Transfer Funds (for inter-bank transfers)
Library Management System

Actors:

  • Student (primary)
  • Librarian (primary)
  • System Administrator (supporting)

Use Cases:

  • Borrow Book
  • Return Book
  • Search Catalog
  • Add New Book (Librarian)
  • Generate Reports (Librarian)
  • Manage Users (Admin)
  • Calculate Fine (included when returning late)
  • Reserve Book (extends Search Catalog)
Common Mistakes to Avoid
  • Don't confuse actors with specific people - actors are roles
  • Don't include implementation details in use case names
  • Don't create too many relationships - keep it simple
  • Don't forget the system boundary
  • Don't make use cases too granular (e.g., "Click Button" is too detailed)

Best Practices

  • Use verb phrases for use case names (e.g., "Register Student", not "Student Registration")
  • Keep use cases at a high level of abstraction - focus on what, not how
  • Ensure every actor is connected to at least one use case
  • Use «include» for mandatory shared behavior
  • Use «extend» for optional or conditional behavior
  • Use generalization to show variations of similar use cases
  • Always include a system boundary to clarify scope
For exams, be prepared to: (1) Identify actors and use cases from a scenario, (2) Draw a complete use case diagram with all relationships, (3) Explain the difference between «include» and «extend», (4) Write user stories for given use cases.
